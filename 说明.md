读完这些文件后，当前系统的运行逻辑可以概括为：

**整体架构**

这是一个“本地/私有知识库 RAG 学习助手”：

- 后端：FastAPI，入口是 [app/api.py](/Users/apple/Documents/php/RAG/app/api.py)
- 前端：Vue 3 + Vite，入口是 [frontend/src/App.vue](/Users/apple/Documents/php/RAG/frontend/src/App.vue)
- 向量库：ChromaDB，配置在 [app/vector_store.py](/Users/apple/Documents/php/RAG/app/vector_store.py)
- 大模型：
  - Ollama 负责本地 embedding 和默认聊天模型
  - DeepSeek 可作为聊天模型后端
- 文档来源：
  - 系统公共知识库：`data/raw`
  - 用户个人知识库：`data/users/<user_id>`
- 用户状态：
  - 账号：`db/users.json`
  - 学习画像：`db/profiles/<user_id>.json`
  - 复习计划：`db/review_sessions/<user_id>.json`
  - 长期记忆：ChromaDB 的 `user_memory_<user_id>` collection

**登录和权限逻辑**

前端首次进入会显示登录/注册页。登录后 token 存在浏览器 `localStorage`。

后端认证在 [app/auth.py](/Users/apple/Documents/php/RAG/app/auth.py)：

- 密码用 PBKDF2-SHA256 加盐存储
- token 是 HMAC 签名的自定义 token
- 默认有效期来自 `.env` 的 `AUTH_TOKEN_TTL_HOURS=168`
- 大多数 API 都要求 `Authorization: Bearer <token>`
- 后端会校验 `request.user_id` 必须等于当前登录用户 ID，避免访问别人数据

**知识库导入逻辑**

上传入口是：

- 前端：[frontend/src/pages/KnowledgePage.vue](/Users/apple/Documents/php/RAG/frontend/src/pages/KnowledgePage.vue)
- 后端：`POST /api/upload`

上传文件支持：

```text
.txt
.md
.pdf
```

流程是：

1. 用户上传文件
2. 文件保存到 `data/users/<user_id>/`
3. 后端调用 [app/ingest.py](/Users/apple/Documents/php/RAG/app/ingest.py) 的 `index_file`
4. 文档被读取：
   - TXT/MD 直接读文本
   - PDF 用 `pypdf` 按页抽取文本
5. 文本清洗、切片
6. 用 Ollama embedding 模型生成向量
7. 写入 ChromaDB 用户专属 collection：`rag_demo_<user_id>`

公共系统知识库的手动重建逻辑是：

```bash
python -m app.ingest
```

它默认扫描 `data/raw`，重建公共 collection：`rag_demo`。

**问答主流程**

核心在 [app/api.py](/Users/apple/Documents/php/RAG/app/api.py) 的 `POST /api/query`。

一次提问的实际链路是：

```text
用户问题
-> 校验登录用户
-> 校验 chat_model 是否允许
-> 检查 Ollama 是否可用
-> 如果有聊天历史，先把问题改写成独立问题
-> 用模型提取核心知识点
-> 拼成增强检索 query
-> 用 embedding 模型向量化
-> 同时检索公共知识库 rag_demo 和个人知识库 rag_demo_<user_id>
-> 合并排序，去掉近邻重复片段，最多取 TOP_K
-> 检索用户长期记忆 user_memory_<user_id>
-> 构造上下文
-> 调用聊天模型生成回答
-> 再生成 3 个推荐追问
-> 抽取本轮对话中值得长期记住的用户事实
-> 写入长期记忆向量库
-> 分析用户学习状态
-> 更新学习画像
-> 返回答案、引用片段、追问、核心词、学习诊断、画像
```

返回给前端的数据大概是：

```json
{
  "answer": "...",
  "chunks": [],
  "followups": [],
  "core_terms": [],
  "learning_insight": {},
  "profile": {}
}
```

前端 [frontend/src/chatStore.ts](/Users/apple/Documents/php/RAG/frontend/src/chatStore.ts) 会把聊天记录存在浏览器 `localStorage`，并缓存学习画像。

**模型逻辑**

配置来自 [app/config.py](/Users/apple/Documents/php/RAG/app/config.py) 和 `.env`：

- 默认聊天模型：`qwen2.5:3b`
- embedding 模型：`bge-m3`
- 可选聊天模型：
  - `qwen2.5:3b`
  - `deepseek-v4-pro`
  - `deepseek-v4-flash`

[app/ollama_client.py](/Users/apple/Documents/php/RAG/app/ollama_client.py) 的规则是：

- `embed()` 永远走 Ollama `/api/embed`
- `chat()` 如果模型名以 `deepseek-` 开头，就走 OpenAI-compatible DeepSeek API
- 否则走 Ollama `/api/chat`

注意：`.env` 里包含 DeepSeek API key，建议不要提交到仓库，也最好轮换一次密钥。

**学习画像逻辑**

每次问答后会调用 [app/learning_analyzer.py](/Users/apple/Documents/php/RAG/app/learning_analyzer.py)：

它会结合：

- 本轮问题
- 回答
- 核心词
- 检索片段
- 用户长期记忆
- 已有学习画像

让模型输出结构化诊断，比如：

- 当前主题
- 核心概念
- 薄弱概念
- 知识漏洞
- 误区
- 推荐复习内容

然后 [app/learning_profile.py](/Users/apple/Documents/php/RAG/app/learning_profile.py) 会把这些合并进用户画像，维护概念薄弱度分数。

**复习计划逻辑**

前端入口是 [frontend/src/pages/ReviewPage.vue](/Users/apple/Documents/php/RAG/frontend/src/pages/ReviewPage.vue)。

后端主要 API：

- `POST /api/review/plan`：生成学习/复习计划
- `POST /api/review/answer`：判断单题答案
- `POST /api/review/complete`：提交整张答卷
- `GET /api/review/history/{user_id}`：读取历史

复习计划会根据 topic、薄弱度和难度生成：

- 讲解模块
- 问答题
- 选择题
- 填空题

提交后会计算正确数，并调整该主题的薄弱度：

- 不及格：提高薄弱度
- 优秀：降低薄弱度
- 及格：保持或轻微更新

**前端页面逻辑**

路由在 [frontend/src/router.ts](/Users/apple/Documents/php/RAG/frontend/src/router.ts)：

- `/guide`：说明页
- `/chat`：知识库问答
- `/knowledge`：知识库管理
- `/review`：学习计划
- `/learning-history`：学习历史
- `/` 自动跳到 `/chat`

前端 API 都集中封装在 [frontend/src/api.ts](/Users/apple/Documents/php/RAG/frontend/src/api.ts)。如果设置了 `VITE_API_URL` 就请求指定后端，否则默认同源请求。

**当前系统一句话总结**

这个系统现在的核心逻辑是：用户登录后上传个人文档，后端把文档切片并写入 ChromaDB；提问时同时检索公共知识库、个人知识库和用户长期记忆，把结果交给 Ollama 或 DeepSeek 生成回答；回答后再更新用户学习画像，并基于薄弱概念生成复习计划。
